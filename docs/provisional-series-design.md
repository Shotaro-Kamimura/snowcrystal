## 2. 「仮」マーキングの三層方式(本設計の柱)

### 2.1 データ層 — 既存フィールドで表現可能(enum 拡張不要)と判定

ソース確定(2026-06-11、HEAD 9f3c368): `DiagramRegion`(src/diagram/types.ts)は `fidelity: 'exact' | 'approx'` / `source: string`(**自由文字列**)/ `confidence: 'high' | 'mid' | 'low'` を持ち、3 フィールドとも dist/index.d.ts に公開済み。`CompositeEntry`(src/growth/types.ts)も同語彙の `fidelity / source / confidence` を持つ(こちらは非公開 — src/index.ts に growth の export なし、d.ts に growth 系シンボルが存在しないことを機械確認済み)。さらに `ml66/P1f` が `source: 'provisional'` の**前例**を既に持つ。

→ **判定: enum 拡張(= d.ts 変更)は不要**。仮実装は既存 3 フィールドの運用規約で表現する:

| フィールド | 規約 |
|---|---|
| `source` | 接頭辞 `'provisional: '` + 仮採用の根拠 + 連動確認事項番号。例 `'provisional: 平行広幅解釈(v1-to-v2 解釈ノート)— 専門家確認(1)待ち'`。**機械可読の仮フラグはこの接頭辞 1 本**(UI 層が導出に使う) |
| `fidelity` | 仮実装中は **'approx' 据え置き**(mlCode と描画の一致が未確証のため)。9 月確定で 'exact' へ昇格。案 M(P1d/P1f/P1f)が approx → exact にしたのは形状定義が文献上一義だったからで、解釈自体が割れている K-a/K-b とは条件が異なる |
| `confidence` | 図上範囲の確度として据え置き(描画の仮実装とは独立 — 案 M の整理を踏襲) |

接頭辞 `'provisional: '` の後ろには**「何が仮か」を必ず書く**(例: P1f = 図上範囲(独自定義・出典なし)/ K-a = 形状解釈(確認(1)待ち)/ K-b = 差分パラメタ(確認(5)待ち))。UI バッジは共通の「仮」でも、データと docs では仮の対象が読み分けられることを保証する。

注: 既存 `ml66/P1f` の裸の `'provisional'`(図上範囲の仮)は、新規約では `'provisional: s 値デジタイズ未確定(羊歯帯)'` 相当へ正規化する(K2 同梱、裁量 2)。「図上範囲が仮」と「形状解釈が仮」は別種の仮であり、source の文章で区別する(フラグは共通)。

### 2.2 UI 層 — 控えめな「仮 / provisional」バッジ

9 月に見せるときの誠実さ(控えめ・事実ベース)の流儀で:

- 情報パネルの形態名の横に小さなバッジ **「仮 / provisional」**(薄い琥珀系・小フォント。`title` 属性に「9 月専門家確認(n)待ちの仮実装 / Provisional — pending expert review」)。3D キャンバス上には出さない。
- 判定は **ML66 データセットから導出**: 現在の RegionHit(または手動形態に対応する region)の `source` が `'provisional'` で始まるか。playground 側に仮形態のハードコード表は**持たない**(単一情報源。LABELS 等の「表示専用コピー」とは性格が異なり、こちらはデータ自体が真実源を持つため)。
- 手動モードの形態セレクト: 仮形態の option テキストに「(仮)」を付す(広幅枝(仮)/ 鱗状側面(仮))。
- 成長パスモード: 複合行は `CompositeEntry.source` から同じ規則で判定。プリセット名は「積乱雲コース(仮)」とラベル自体に明示(§6)。

### 2.3 docs 層 — 仮実装台帳(本書 §8)

本書 §8 に**台帳**を置き、各ストリームの 仮採用値 / 連動確認 / 確定時手順 / 修正時手順 を 1 行ずつ管理する。運用規則:

- 各 CP クローズ時に台帳の該当行を更新(採用値・コミットハッシュ)。判定記録はコミット本文に永続化(c40ec0a・760ddeb 方式)。
- 9 月の回答後: **確定** → source から 'provisional: ' を外し fidelity を 'exact' へ、台帳行に確定日付と回答要旨を記録。**修正** → 台帳行の「修正時手順」列に列挙したファイルだけを触れば差し替えが完結することをこの設計で保証しておく(各ストリーム節に差し替え範囲を明記)。
- どちらの場合も d.ts への影響は「union 値の改名があるときのみ」発生(その場合は案 M 基線手順を再適用)。

## 3. K-a: 広幅枝 P1c 専用ジオメトリ(仮)

### 3.1 根拠と解釈の割れ目

- ML66 P1c = crystal with broad branches(広幅の枝を持つ結晶)。Fig.2 では (−17, −13] バンドの P1b(扇形)と P1d(星状)の間(sTop 0.85–1.0。ml66 設計書 §5・2026-06-10 自己監査でスケッチ同定済み)。現状は `morphology: '扇形'` の approx(src/diagram/ml66.ts)。
- v1-to-v2 解釈ノート(扇形 P4f): 「扇形は**平行側面(広幅型)**の解釈で実装。**拡幅セクター型**(中心から外周へ向かって幅が広がる形)のモデル化は専門家確認事項」 — 確認 (1) の実体。つまり「広幅」と「セクター」のどちらが P1b でどちらが P1c かという割り当てが未確定。

### 3.2 仮形状 2 案(両方実装して 9 月に並べる)

## 4. K-b: 側面 S2 差分ジオメトリ(仮)

### 4.1 S2 の定義(リポジトリ内ノート)

- ml66 設計書 F8: 「S2(**鱗状側面**)・S3 は −25〜−35°C(Weickmann)」。sideplanes 設計書 §1: 「S2 は**小さな鱗状板の重なり**」(ML66 Fig.1 読解)。同 §0・検収論点 4 で「S2 = 同一レンダラ approx」を当時追認 → 本ストリームでその差分を仮実装する。
- 確認 (5) の本体である**縦位置**(ML66 忠実 = 水飽和の上 vs MKY71 下方修正)は**動かさない**。K-b は「S1 と S2 が画面で別物として見える」足場を作り、(5) の議論を具体化する。

### 4.2 仮差分(側面のパラメタ族)

`createSidePlanes`(src/geometry/parts.ts)のパラメタ族として実装。フィン形状(半六角・スパイン = a 軸)と CSL 双晶角 70.3° アンカーは**不変**(S2 も同族の多結晶側面 — 結晶学アンカーは仮にしない):

- フィン小型化: `radiusBase` 0.9 → 0.5 前後(小さな鱗)。
- 枚数増: count 6〜7(現行レンジ [4,7] の上側に固定)。
- **重なりの強調**: スパイン方向スタッガ係数 ±0.4 → ±0.9 前後(`SidePlanesOptions` に `staggerSpan` を追加 — parts.ts は非公開モジュールのため d.ts 影響なし)。鱗が軸方向にずれて重なる Fig.1 の読みを表す。
- 最終値は実装時目視で確定(砲弾・側面 S1 と同方式の一任)。

### 4.3 union 判定

- **「側面」のままでは済まない**と判定: K-a §3.3 と同じ構造(公開経路に region 別バリアントなし)により、S2 を S1 と描き分けるには union 値が必要。**'鱗状側面'** を追加(15 → 16 値)。
- ml66.ts: `ml66/S2` を `morphology: '鱗状側面'`・labelJa '鱗状側面'・fidelity 'approx' 据え置き・source 'provisional: ' 規約へ。TITLE_MAP 1 エントリ(案: 鱗状側面 Scalelike Side Planes)。コード割当は ''(未割当)。
- d.ts: 基線 #4 → #5(案 M 手順の再適用)。
- 9 月差し替え範囲: 確定 → ml66.ts のみ。「S1 と同形で良い」への修正 → ml66/S2 の morphology を '側面' へ戻す + union から '鱗状側面' を除去(d.ts 縮小 = 基線再更新)+ 台帳記録。縦位置の修正(MKY71 採用)→ ml66.ts の bands(S2/C2a の sTop)のみで、ジオメトリと独立。

## 5. K-c: P2 系複合ジオメトリ(仮)

### 5.1 方式の選定: 「枝先・頂点に終端要素を付ける」コンポジット

- 対象: COMPOSITE_TABLE(src/growth/composites.ts)の **P2a/P2c**(branched → plate: 角板付六花・樹枝)と **P2f/P2g**(plate → branched: 扇・樹枝付角板)を classification-only から geometric へ。
- 構成 = **中核形態(ステージ①)+ 終端部品(ステージ②)× 6**:
  - P2a/P2c: 中核 = ステージ①の branched 形態 + **小六角板**(CylinderGeometry(…, 6))を 6 腕の先端に。中核に `pathHit.stages[0].morphology` を使うことで、①が星状なら P2a・樹枝状なら P2c の区別が**自動で**出る(対応表の 'P2a/P2c' 併記と整合)。
  - P2f/P2g: 中核 = 六角板 + 6 頂点に終端枝 — 扇形花弁(P2f)/ 樹枝腕(P2g)— ②の morphology で選ぶ。
- **枝先端座標は方位規約 R1〜R7 からの計算のみ**(案 N の実証済み方針 — メッシュ走査に依存しない): 樹枝族の腕先端 = φ = 30° + k·60°・r = mainLength 2.1(DENDRITE_ARM_DEFAULTS)/ 扇形・広幅枝の花弁先端 = 同方位・r = 0.42 + 1.1 = 1.52 / 角板頂点 = 同方位・r = 板外接半径。終端部品も Cylinder 族(30° 位相)で格子整合を保つ。具体値の確定は K3(§5.3)。
- **塗り分け**: 各 Mesh に `userData.part: 'core' | 'tip'`(冠柱の 'column' | 'cap' 前例)。playground の `paintGrowthParts` を part → ステージ色のマップに一般化(① = core / ② = tip)。パッケージのマテリアルは従来どおり単色。
- データ: `CompositeMorphology` を `'冠柱' | '角板付枝' | '樹枝付角板'` へ拡張し、該当 4 行の morphology を非 null 化・source を 'provisional: ' 規約へ(fidelity 'approx' 据え置き)。renderGrowthPath のディスパッチに 2 case 追加。
- **公開面への影響なし**: growth は src/index.ts に export されておらず d.ts に growth 系シンボルが存在しないことを確認済み → **d.ts 基線不変・dist のみ新基線**というゲート上の好性質を持つ。

### 5.2 CP1b への応用可否

**可**と判定: 同じ「終端要素」機構で、中核 = 角柱・終端 = 樹枝星(基底面 ⊥ c 軸、y = ±L/2 の両端面に配置)とすれば CP1b(樹枝付柱)は冠柱のキャップ差し替えとして実装できる(createCappedColumn の構成と R1 規約がそのまま使える)。採否と範囲(K-c 同梱か外すか)は **K3 で確定**(工数を見て判断 — 裁量 6)。

### 5.3 詳細設計は CP-K3 に送る

寸法(終端板の R・厚み、終端枝のスケール)・干渉判定(終端部品同士/中核との重なり)・テスト計画(ゴールデンパス・userData.part・先端座標の機械検証)・CP1b 採否は、専用の設計 CP(CP-K3)で確定する。**K3 の停止構成**: 設計のみ・docs 1 ファイル(本書 §5 の具体化、または本書への追補節)・ゲート = 全定型 + dist/d.ts 基線不変 → ドラフト全文をチャット報告 → 検収後にコミット(本 CP-K1 と同形)。
