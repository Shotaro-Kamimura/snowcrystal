## 2. 縦軸の設計判断: 相対飽和座標 s

採用案: 内部座標として **相対飽和 s = ρ / ρ_ws(T)** を導入する。

- ρ = 氷飽和に対する過剰水蒸気密度 [g/m³](現行スライダー値。v1 の 0–0.3 軸の実体)
- ρ_ws(T) = 水飽和と氷飽和の飽和水蒸気密度差 [g/m³](温度のみの物理曲線)
- したがって s=0 が氷飽和、**s=1 が水飽和**、s>1 が雲粒域

理由: ML66 の領域は「氷飽和近傍」「水飽和線上」「水飽和の上」という**アンカー相対**で記述されている(F1, F4, F10)。s 座標はこのアンカーをそのまま数値化したもので、原典に対して最も忠実。さらに水飽和線の曲率(温度依存)は ρ_ws(T) が物理として担うため、**データセット側の境界値は区分線形の s 値だけで済む**。

公開 API・スライダーは現行どおり ρ(0–0.3)を維持し、lookup 内部で s に変換する。後方互換と、Nakaya 図(v1)との見た目の連続性のため。

ρ_ws(T) の実装方針: Murphy & Koop (2005) の飽和蒸気圧式(水・氷)から ρ = e/(R_v·T) で差を取る。サニティ値: ρ_ws(-15°C) ≈ 0.21–0.22 g/m³、極大は -12〜-15°C 付近 ≈ 0.22–0.23 g/m³。現行スライダー上限 0.3 はこの極大を覆っており、**v1 の暗黙の軸選択と整合する**(実装時に式とテーブルでテスト確定)。

留意点: 低温では ρ_ws が小さく、スライダー 0.3 は s≫1 に達する。物理的にはリム域だが、本パッケージは形態可視化であり着氷は描かないため、**s が最上段領域の下限を超えた場合は最上段の結晶形を返す**(割り切りを docs に明記)。

## 3. スキーマ(TypeScript、THREE 非依存)

配置: `src/diagram/`(`crystallography.ts` と同格の純粋モジュール)。

```ts
// src/diagram/types.ts
export type DiagramId = 'nakaya-v1' | 'ml66';

/** ML66 のタイプコード(N1a, C1c, P1e, S1 など)。Kikuchi グローバル分類とは別名前空間(§6) */
export type MlCode = string;

export interface DiagramRegion {
  id: string;                    // 'ml66/P1e' のように diagram を前置
  mlCode: MlCode | null;         // 出典コード(nakaya-v1 では null)
  morphology: Morphology;        // 描画キー(v2 で拡張、§6)
  fidelity: 'exact' | 'approx';  // mlCode と描画形態の一致度
  labelJa: string;               // 図面ラベル(例 '樹枝状')
  source: string;                // 'ML66 §3.1' | 'ML66 Fig.2(デジタイズ)' | 'MKY71' | 'provisional'
  confidence: 'high' | 'mid' | 'low';
}

/** バンド内の縦積み1段。sTop は [tMax(高温端)での上限s, tMin(低温端)での上限s](T について区間線形)。最上段は省略=∞ */
export interface StackEntry { regionId: string; sTop?: readonly [number, number]; }

/** 温度バンド。T ∈ (tMin, tMax](現行 getCrystalType の半開区間規約と同じ向き) */
export interface TemperatureBand { tMax: number; tMin: number; stack: readonly StackEntry[]; }

export interface ConditionDiagram {
  id: DiagramId;
  vaporCoord: 'rho' | 's';       // nakaya-v1 は 'rho'(現行しきい値の忠実移植)、ml66 は 's'
  tDomain: readonly [number, number];    // [-40, 0]
  rhoDomain: readonly [number, number];  // [0, 0.3]
  regions: Readonly<Record<string, DiagramRegion>>;
  bands: readonly TemperatureBand[];     // tMax 降順・隙間なく tDomain を被覆
}
```

## 4. ルックアップ仕様と境界規約

```ts
// src/diagram/lookup.ts
export interface RegionHit { region: DiagramRegion; morphology: Morphology; mlCode: MlCode | null; }
export function classifyOnDiagram(tempC: number, vapor: number, diagram?: ConditionDiagram): RegionHit;
```

- バンド選択: T ∈ (tMin, tMax]。T=0 は最暖バンドに含める。tDomain 外はクランプ。
- 縦選択: vaporCoord='s' なら s = vapor / ρ_ws(T) に変換してから、下から順に **s ≤ sTop(T)**(線形補間)を満たす最初の段。境界値は下側の段に所属(現行 `getCrystalType` の ≤ 規約と一致させ、v1 回帰を厳密一致にするため)。最上段は sTop なし=無限大(§2 の割り切り)。
- 全域被覆・決定性: バンドが tDomain を隙間なく覆い、各バンド最上段が無限大であることをスキーマ不変条件としてテストで保証(§9)。乱数なし。

## 6. 形態マッピングと新形態

### 6.1 コードの名前空間衝突(重要)

ML66 コードと、本パッケージが既に UI で使っている **Kikuchi ほか(菊地ほか 2012 / Kikuchi et al. 2013)グローバル分類**のコードは、**同じ表記で別の意味**を持つ(例: C1c = ML66 では中実砲弾 / グローバル分類では針集合。C2a = ML66 では砲弾集合 / グローバル分類では鞘)。混線は学術的信頼性に直結するため、型レベルで分離する:

- `mlCode`(ML66 名前空間)と `globalCode`(グローバル分類名前空間)を別フィールドとして保持し、裸の文字列を共有マップに混ぜない
- UI は将来「ML66: P1e / Global: P4g」の併記が可能な構造にしておく(表示変更自体は Phase 2 スコープ外)

### 6.3 新形態の確定案(「3〜4種」の解)

確定 3 種: **砲弾**(C1c/C1d 統合)・**砲弾集合**(C2a)・**側面**(S1。S2/S3 は同形態の approx 描画)。

第 4 候補: **長柱**(N1e)。代替案として「角柱のアスペクト比パラメータで代用」があり得るが、F3 のとおり N1e は温度条件が固有(-30°C 以深)で図上の独立領域になるため、領域としては必ず立てる。**描画を新規にするか角柱の細長変形で済ませるか**が検収論点 3。

結晶学メモ(実装フェーズ向け・1 行ずつ): 砲弾=六角柱+錐状終端(角錐面は {10-1̄1} 系トレースで閉じる)。砲弾集合=共通核から放射する複数砲弾(凍結雲粒起源の多結晶)。側面=c 軸に平行な薄板の集合体。いずれも `crystallography.ts` の流儀(アウトライン関数+角度テスト)で実装する。形態の幾何精度は従来どおり MAX 工数で扱う。

## 7. 履歴型の外出し(Phase 3 への接続)

F7 のとおり、P2(異形枝)・P5(空間枝)・P6(放射)・CP(柱・板の複合)は ML66 自身が「落下中の環境変化の記録」と位置付けるタイプであり、**単一の (T, s) 点では原理的に生成できない**。したがって ml66 図の領域にはせず、Phase 3(環境履歴/成長パス UI)の入力仕様として引き継ぐ。これは 9 月の雪氷研究大会ポスター・BGR 骨子の「複合型=環境履歴の記録」構想の実装面の根拠と一致する(一次根拠が ML66 §3.4–3.7 に直接ある、と骨子に書ける)。R 系(着氷)は雲粒過程であり本パッケージの範囲外として明記する。

## 9. テスト計画(vitest、既存の角度テストと同居)
